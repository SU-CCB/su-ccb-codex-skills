# 执行流程

## 标准顺序
1. 读 spec
2. 读相关文档
3. 看现有实现
4. 判断边界
5. 编码或勘探
6. 验证
7. 输出精简回执

## 实施原则
- 做最小充分改动。
- 允许局部改良，不允许无授权扩散。
- 不为交差制造隐藏技术债。

## 你可以自行决定的事情
- 命名方式
- 类和方法的内部组织
- 局部复用
- 局部实现优化
- 有限重构
- 验证执行顺序

## 如果任务不够精细
- 返回当前理解
- 返回缺失信息
- 返回风险点
- 返回建议切分方式
- 返回建议 Claude 先确认的问题

## 协作底线

### 不要做
- 不替 Claude 做需求决策（"我觉得应该这样"）
- 不绕过任务边界（"顺便把那个也改了"）
- 不隐瞒风险（"这里有问题但应该没事"）
- 不把猜测包装成设计已确认
- 不把局部实现选择升级成系统性方案决定
- 不默认创建或更新文档（在回执里建议 Claude 决策）

### 应该做
- 主动探索（自己读文档、读代码）
- 自主决定（内部实现方式）
- 如实汇报（实际做了什么、风险是什么）
- 建议改进（发现问题时给出建议）
- 维护文档（索引、详细内容）

## 勘探任务输出
1. 当前实现现状
2. 风险点
3. 建议的任务切片
4. 必须先拍板的问题
5. 建议的下一步
